# Functional Specification — Template

Write the document as `functional-specification.md` using exactly the
Markdown subset `scripts/build_docx.py` supports: headings `#`–`####`,
paragraphs with `**bold**`/`*italic*`/`` `code` ``, `-`/`1.` lists, `|`
tables, `![caption](path)` images, ``` code blocks, `>` callouts. Image
paths are relative to the `.md` file (`diagrams/01-context.png`).

Text in [square brackets] is instructions to you — replace it with real
content. Skip a subsection only when the repository genuinely has nothing
for it, and say so in one line rather than deleting the heading silently.

The FS describes **observable behavior**: a tester should be able to derive
test cases from it without reading the code. Keep design details (class
names, DI, EF Core) out of it — those belong in the MDS. Referring to
routes, field names, and error messages is encouraged; referring to
`OrderService.PlaceOrderAsync` is not (cite files only in evidence notes).

## ID conventions (used for cross-referencing, keep them consistent)

- Feature areas: `FA-1`, `FA-2`, … — one per `##` section in chapter 4.
- Functional requirements: `FR-<area>-<n>`, e.g. `FR-2-3` = third
  requirement of feature area 2.
- Business rules: `BR-<n>` — global catalog in chapter 5.

## Frontmatter (required — drives the cover page)

```
---
title: [System name]
subtitle: Functional Specification
project: [Full project/solution name]
version: 1.0
date: [today, yyyy-MM-dd]
author: [Generated by <agent> / team name if known]
status: Draft
---
```

## Section skeleton

```
# Introduction

## Purpose
[2–3 sentences: this document specifies the functional behavior of <system>,
reverse-engineered from the source code at a point in time.]

## Scope
[Which solution/projects are covered and to what depth; what is out of scope
(UI pixel design, external systems' behavior, performance testing).]

## Audience
[Business analysts, testers, product owners, developers new to the system.]

## Definitions and Acronyms
| Term | Definition |
[Domain terms and acronyms used below — from entity names, README, code.]

# System Overview

[One or two paragraphs: what the system does in business terms and who uses
it. Then the context diagram.]

![System context](diagrams/01-context.png)

# Actors and Roles

| Actor / Role | Description | How authenticated |
[From auth schemes, [Authorize] roles/policies, seeded roles. If endpoints
are unauthenticated, one row: "Anonymous user" — and say so plainly.]

# Functional Requirements

[Intro sentence + the feature map figure, then one `##` per feature area.]

![Feature map](diagrams/02-feature-map.png)

## FA-1: <Feature area, e.g. Order Management>

[One-paragraph summary of the capability and who uses it.]

### FR-1-1: <Requirement name, e.g. Place an order>

[For each requirement — typically one per endpoint/operation:]

**Trigger:** [route + verb, scheduled time, or message that starts it]

**Inputs:**
| Field | Type | Constraints |
[From the request DTO + validators: required, ranges, formats. Include the
actual error message where the validator defines one.]

**Processing:**
1. [Numbered, observable steps in business language: "the stock level of
   each ordered product is checked", "the card is charged via the payment
   provider". Include conditions and branches.]

**Outputs:** [what the caller gets on success — status code, payload
essentials — and what state changed / which notifications were sent.]

**Errors:**
| Condition | Behavior |
[Each validation failure, business-rule rejection, or exceptional path the
code actually handles, with the observable result (status code, message).]

[Repeat FR-1-2, FR-1-3, … then FA-2, FA-3, …]

# Business Rules

| ID | Rule | Enforced in |
[Global catalog: every domain invariant, guard clause, and cross-cutting
validation found. Rule in business language ("an order below the minimum
amount of 5.00 cannot be placed"); "Enforced in" = file/member, so
developers can trace it. Reference these IDs from FR Processing steps.]

# Key Process Flows

[For 1–3 core business processes, a short narrative plus the flow diagram.]

![Process — <name>](diagrams/03-process-<flow>.png)

# Entity Lifecycles

[When a central entity has a status/state field: narrate the lifecycle and
who/what causes each transition.]

![State model — <entity>](diagrams/04-state-<entity>.png)

# External Interfaces

| System | Direction | Purpose | Data exchanged |
[Third-party APIs, message brokers, email/SMS, file storage — from the
functional viewpoint (what business purpose, what data), not the technical
client classes.]

# Data Requirements

| Entity | Purpose | Key information held |
[Business-level view of the entities: what real-world thing each represents
and the information the system must retain about it. No column types here —
the MDS owns physical design.]

# Non-Functional Observations

[Only what the code evidences: authentication/authorization coverage, input
validation coverage, auditing/logging of business events, rate limiting,
localization. Honest gaps included ("no audit trail of catalog changes was
found").]

# Assumptions and Open Questions

- [Anything inferred rather than read, behavior that looks unintended,
  endpoints whose purpose is unclear. Number them so reviewers can respond.]

# Appendix A — Requirement Traceability

| Requirement | Entry point | Feature area |
[One row per FR: FR id, route/verb (or job/message), FA id. This is the
tester's index into the system.]

# Appendix B — Diagram Sources

[Table mapping each figure to its editable source in diagrams/ — .mmd
(re-render with render_diagrams.py) and .drawio where produced.]
```
